# Bible Journey: implementación integral de producto, diseño y motor editorial

Actúa como Lead Product Engineer, Mobile UX Architect y Content-Platform Engineer para Bible Journey.

No partas de cero ni reemplaces funcionalidad útil. Primero inspecciona el proyecto actual, identifica la arquitectura, pantallas, navegación, modelos, estado y componentes ya existentes. Conserva lo que sea reutilizable, pero refactoriza sin miedo lo que contradiga esta especificación.

Tu trabajo no es crear una demo visual. Debes convertir el avance actual en una aplicación funcional, escalable y editorialmente seria.

Los archivos adjuntos son obligatorios como fuente de verdad:

1. `Bible_Journey_Master_Canon_Ledger_v1_0.xlsx`
2. `Bible_Journey_Harmonization_Engine_Specification_v1_0.docx`

Estos archivos definen la arquitectura editorial, los mapas canónicos, los bloques de lectura, los vínculos entre libros, los perfiles cronológicos, las decisiones abiertas y las reglas del Harmonization Engine.

No simplifiques esos documentos a una ruta demo de David. La visión es una Biblia completa, navegable, cronológica, honesta con la incertidumbre y agradable de leer.

# 1. Objetivo del producto

Bible Journey debe permitir que una persona lea toda la Biblia de manera continua, pero organizada por acontecimientos históricos, relaciones literarias, relatos paralelos, profetas, cartas, poesía y contexto.

La idea central es:

> La Biblia completa es la columna vertebral.
> Bible Stream organiza la lectura.
> Journeys ofrece lentes narrativos y temáticos.
> Study Mode profundiza.
> Ezra responde preguntas con fuentes y niveles de certeza.

La aplicación no debe sentirse como:

* una Biblia PDF;
* una colección de tarjetas;
* una app de cursos;
* una app de IA con textos bíblicos dentro;
* una armonía artificial que mezcla versículos de varios libros.

Debe sentirse como una experiencia moderna, editorial y contemplativa para seguir la historia bíblica sin perderse.

---

# 2. Regla editorial principal

Nunca fusionar, reescribir o intercalar versículos de varios libros para construir una narración artificial.

Ejemplo correcto:

```text
Acontecimiento: Senaquerib amenaza Jerusalén

1. 2 Reyes 18:13–19:37
   Relato principal

2. Isaías 36–37
   Relato paralelo profético

3. 2 Crónicas 32:1–23
   Relato complementario
```

Ejemplo incorrecto:

```text
Versículo de Reyes
+
Versículo de Isaías
+
Versículo de Crónicas
=
Una narración nueva mezclada por la aplicación
```

Cada bloque bíblico debe mantenerse intacto, identificado por libro, referencia, traducción, licencia, rol editorial, fuente y nivel de certeza.

La aplicación organiza la lectura. No reescribe la Escritura.

# 5. Temas de lectura

Implementa dos temas equivalentes, ambos accesibles desde preferencias y desde el lector.

## A. Modern Deep

Usar para:

* Home;
* Journey;
* Ezra;
* exploración;
* progreso;
* mapas;
* configuración;
* navegación principal;
* premium.

Características:

* fondo azul carbón;
* acentos violeta controlados;
* tarjetas mínimas;
* sombras discretas;
* tipografía sans limpia.

## B. Editorial Light Reader

Usar principalmente para lectura prolongada.

Características:

* fondo marfil muy claro;
* texto casi negro;
* tipografía serif para Escritura;
* acentos bronce oscuro o violeta sobrio;
* alto contraste;
* sin aspecto vintage;
* lectura cómoda bajo sol y para usuarios mayores.

No llamar esto “modo claro” y “modo oscuro” en el producto principal.

Usar una nomenclatura editorial, por ejemplo:

```text
Modern Deep
Editorial Light
```

El usuario puede elegir su preferencia y la aplicación debe recordarla.

# 7. Pantallas y flujo visual

## A. Home

La pantalla Home debe tener foco en continuar.

Arriba del fold solo deben aparecer estas tres secciones:

1. Continue Reading
2. Journey Progress
3. Today’s Connection

No llenar el primer viewport con catálogos o demasiadas rutas.

### Continue Reading

Debe mostrar:

```text
The Life of David
Act I · The Chosen One

Samuel anoints David
1 Samuel 16:1–13

Continue Reading
```

También debe poder mostrar el contexto del Bible Stream:

```text
Bible Stream
United Monarchy
1 Samuel 16:1–13
```

El usuario debe entender que puede continuar desde dos perspectivas distintas del mismo progreso:

```text
Bible Stream
La secuencia bíblica completa

Journey
La ruta narrativa actual
```

No son lecturas duplicadas.

Son dos lentes sobre el mismo contenido.

### Journey Progress

Mostrar progreso narrativo, no solo porcentaje:

```text
The Chosen One
The Warrior
The Fugitive
The King
The Fall
The Legacy
```

Mostrar dónde está el usuario sin sobrecargar.

### Today’s Connection

Ejemplo:

```text
Today’s Connection

David is still a shepherd when Samuel anoints him.
This image later echoes through Psalm 23.

Explore the connection
```

## F. Study Mode

Study Mode no es Ezra.

Study Mode debe ser una capa editorial para profundizar en el Reading Set actual.

Encabezado:

```text
Samuel anoints David
1 Samuel 16:1–13 · c. 1025 BC · Bethlehem
```

Tabs:

```text
Overview
Context
People
Connections
Notes
```

Abrir por defecto en Overview.

Contenido inicial:

```text
What happened
Samuel privately anoints David while Saul is still king.

Why it matters in the story
This begins David’s long and dangerous path toward the throne.

Explore deeper
People · Location · Related Psalm · Timeline
```

Reglas:

* progressive disclosure;
* no abrir diez tarjetas al mismo tiempo;
* Time, Place, Chronology y Confidence deben ser una franja compacta;
* no usar chat aquí;
* no imponer aplicaciones espirituales;
* cada nota histórica debe tener fuente y nivel de certeza;
* Notes debe permitir guardar reflexión, versículo, pregunta o bookmark.

## G. Ezra

Ezra debe ser una experiencia separada.

Puede abrirse como pantalla o como bottom sheet contextual desde el Reader.

No debe ser un chat genérico.

Debe saber:

* Reading Set actual;
* versículo o bloque activo;
* Journey actual;
* idioma;
* traducción;
* perfil cronológico seleccionado;
* contexto editorial permitido;
* notas o preguntas previas del usuario, si existen.

Abrir con sugerencias contextuales:

```text
Why was Samuel afraid of Saul?
Who was Jesse?
Why was David chosen?
What happens next?
Show related Psalms
```

Cada respuesta de Ezra debe tener estructura visible:

```text
Direct answer

Biblical basis

Historical context

Editorial inference, if any

Confidence

See sources
```

Reglas para Ezra:

1. No presentar inferencias como texto bíblico.
2. No presentar una fecha debatida como hecho.
3. No inventar conexiones entre Salmos y eventos.
4. No usar un tono de predicación automática.
5. No terminar cada respuesta con “¿qué te enseña esto?”.
6. Solo ofrecer reflexión personal si el usuario la pide.
7. Diferenciar visualmente:

   * Biblia;
   * contexto histórico;
   * inferencia editorial;
   * interpretación;
   * aplicación.
8. No generar una cronología nueva por razonamiento libre.
9. Consultar el Harmonization Engine y los datos editoriales aprobados.
10. Si algo está debatido, decirlo claramente.

# 8. Arquitectura editorial: Bible Stream

Bible Stream no se organiza solo por libros.

Se organiza por:

```text
Era
→ Chronological Reading Set
→ Reading Block
→ Biblical Passage
```

Ejemplo:

```text
Era:
United Monarchy

Chronological Reading Set:
Solomon dedicates the temple

Reading Blocks:
1. 1 Kings 8:1–66
   Primary account

2. 2 Chronicles 5:2–7:22
   Parallel / complementary account
```

Otro ejemplo:

```text
Era:
Kingdom of Judah

Chronological Reading Set:
Sennacherib threatens Jerusalem

Reading Blocks:
1. 2 Kings 18:13–19:37
   Primary account

2. Isaiah 36–37
   Parallel prophetic account

3. 2 Chronicles 32:1–23
   Complementary account
```

# 9. Tres modos de lectura

## A. Complete Chronological Reading

Debe leer todos los bloques requeridos de un Reading Set antes de marcarlo como completamente terminado.

Ejemplo:

```text
2 Kings 18–19
→ Isaiah 36–37
→ 2 Chronicles 32
→ Next event
```

No marcar como completo un evento si el usuario dejó relatos paralelos obligatorios pendientes.

Mostrar:

```text
Event fully complete
All related biblical accounts read
```

---

## B. Narrative Flow

Debe permitir seguir la historia con menos repetición inmediata.

Ejemplo:

```text
2 Kings 18–19

This event also appears in:
Isaiah 36–37
2 Chronicles 32

Continue the story
Save related accounts for later
```

Pero no mentir.

Mostrar claramente:

```text
Story progress complete
Main account read

Expanded Bible reading
2 related accounts pending
```

---

## C. Compare Accounts

Debe permitir comparar relatos relacionados sin mezclar texto.

Puede incluir:

```text
2 Kings
Isaiah
2 Chronicles
```

Debe soportar:

* lectura individual;
* switching por cuenta;
* referencias relacionadas;
* diferencias editoriales;
* pasajes exclusivos;
* mapa;
* notas;
* fuente;
* certeza;
* comparación visual futura lado a lado.

No implementar una fusión de versículos.

# 13. Harmonization Engine

Implementar el motor como una capa determinista y auditable.

No usar LLM para decidir el orden de la Biblia.

El LLM puede explicar datos aprobados. No puede crear relaciones editoriales ni reordenar bloques por su cuenta.

El engine debe:

1. recibir Reading Blocks;
2. recibir Relationship Edges;
3. aplicar un Chronology Profile;
4. construir un Stream Plan versionado;
5. producir nodos ordenados;
6. identificar conflictos;
7. registrar certeza;
8. registrar rationale;
9. exponer qué es obligatorio en Complete Mode;
10. exponer qué puede diferirse en Narrative Flow;
11. exponer grupos para Compare Accounts.

El engine debe poder clasificar relaciones:

```text
before
after
parallel_account
complementary_account
prophetic_context
poetic_connection
genealogical_context
legal_context
epistolary_context
canonical_fallback
literary_sequence
context_only
```

Reglas obligatorias:

* nunca crear ciclos;
* nunca publicar un Stream Plan con conflicto no resuelto;
* nunca ocultar una relación debatida;
* nunca mover Apocalipsis a una línea de eventos mundiales;
* nunca forzar Job, Proverbios, Eclesiastés o Cantares a una fecha exacta si el ledger no lo aprueba;
* nunca contar un bloque pendiente como leído;
* nunca marcar todos los relatos paralelos como leídos por leer uno.

# 14. Perfiles cronológicos

Implementar perfiles editoriales configurables.

El perfil por defecto debe ser:

```text
Cautious Default
```

Debe preferir:

* transparencia;
* canonical fallback cuando sea necesario;
* etiquetas claras;
* no afirmar como hecho lo debatido;
* no imponer una tradición específica.

Soportar más adelante:

```text
Traditional Ecclesial
Research Comparison
```

Los perfiles pueden cambiar:

* orden;
* visibilidad de alternativas;
* notas;
* relación entre cartas y Hechos;
* tratamiento de Gálatas;
* cartas de prisión;
* cartas pastorales;
* lentes de interpretación opcionales.

Los perfiles nunca cambian:

* texto bíblico;
* referencias;
* contenido de los pasajes;
* derechos de traducción;
* historial real de progreso del usuario.

# 16. Validaciones y QA editorial

Crear tests automáticos y comandos administrativos para validar:

```text
1. Every canonical chapter appears once in the active canon profile.
2. No Reading Block has invalid references.
3. No passage range overlaps incorrectly.
4. No verse range is omitted in split chapters.
5. No relationship edge creates a cycle.
6. Every debated placement includes a rationale and source.
7. Every visible context card has a source.
8. Every public Stream Plan has an approved version.
9. Narrative Flow does not auto-complete deferred blocks.
10. Complete Chronological Reading requires all required blocks.
11. Compare Accounts never renders merged Scripture.
12. Protected translations never render full text without valid license configuration.
```

Crear un dashboard editorial o admin panel con:

* import batches;
* validation errors;
* unresolved decisions;
* missing sources;
* coverage status;
* affected Stream Plans;
* profile-specific conflicts;
* publish / unpublish workflow;
* audit log.

# 17. Implementación de Ezra

Ezra debe usar retrieval desde datos aprobados.

Contexto mínimo enviado a Ezra:

```text
- current Reading Set
- current Reading Block
- current passage
- selected translation
- selected chronology profile
- user language
- approved editorial notes
- related sources
- certainty labels
- relevant Journey context
```

Formato de respuesta esperado:

```json
{
  "direct_answer": "",
  "biblical_basis": [
    {
      "reference": "",
      "summary": ""
    }
  ],
  "historical_context": "",
  "editorial_inference": "",
  "confidence": "High | Probable | Debated | Unresolved",
  "sources": [],
  "optional_reflection": null
}
```

No permitir que Ezra entregue una respuesta final sin `confidence` cuando hable de cronología, autoría, ubicación o relación entre textos.

# 22. Restricciones finales

No hagas esto:

* no conviertas todo en cards;
* no borres el lector continuo;
* no uses un chat como pantalla de estudio;
* no pongas cinco tabs inferiores;
* no cuentes texto no leído como leído;
* no mezcles relatos paralelos;
* no inventes cronologías;
* no escondas incertidumbre;
* no trates Apocalipsis como cronología mundial por defecto;
* no muestres NVI, NIV o RVR60 completos sin licencia;
* no conviertas Ezra en un predicador automático;
* no bloquees toda la app si una traducción está en Reference Mode;
* no reemplaces las pantallas actuales sin migrar flujos útiles;
* no dejes datos importantes hardcodeados en widgets.

Haz esto:

* prioriza la lectura;
* conserva Escritura intacta;
* separa lectura, estudio, Journey y Ezra;
* registra fuentes;
* registra certeza;
* registra decisiones;
* aplica perfiles;
* calcula progreso con honestidad;
* mantén una experiencia hermosa y simple para el lector.

No necesito una propuesta abstracta. Ejecuta la implementación, documenta decisiones y deja el proyecto en un estado que pueda continuar un equipo real de producto, editorial, backend y mobile.
